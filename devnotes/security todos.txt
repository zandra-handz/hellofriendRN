SECURITY TO-DO CHECKLIST
=========================

GOOGLE API KEY RESTRICTIONS
-----------------------------
The Google API key is embedded in the app bundle (unavoidable for native SDKs
and client-side API calls). Moving it to process.env only protects it from
git exposure -- a determined attacker can still extract it from the built APK/IPA.
The real fix is restricting the key in Google Cloud Console so it only works
when called from your specific signed app.

Steps:
[ ] Go to Google Cloud Console > APIs & Services > Credentials
[ ] Select your API key (AIzaSyAY...)
[ ] Under "Application restrictions":
      - Add Android app restriction:
          Bundle ID: com.badrainbowz.sampleproject
          SHA-1 signing certificate fingerprint (get from EAS or keystore)
      - Add iOS app restriction:
          Bundle ID: com.badrainbowz.sampleproject (or whatever your iOS bundle ID is)
[ ] Under "API restrictions":
      - Restrict to only the APIs you actually use:
          - Maps SDK for Android
          - Maps SDK for iOS
          - Places API
[ ] Save and test that the app still works after restricting

WHY THIS MATTERS
-----------------
Once restricted, the key is useless to anyone who extracts it from your bundle
because it only works for signed requests from your specific app. This is what
Google recommends for mobile apps -- a backend proxy is the "proper" solution
but is overkill for client-side Maps/Places usage.

NOTES
------
- app.config.ts: passes key to native Maps SDK -- cannot be proxied, key restriction is the only fix
- SearchBarGoogleAddress.jsx: passes key to react-native-google-places-autocomplete HTTP calls
    -- could theoretically be proxied through your backend in the future, but
    -- key restriction is sufficient for now
- Both files now read from process.env.GOOGLE_API_KEY (via @env) so the key
    is at least out of git history
